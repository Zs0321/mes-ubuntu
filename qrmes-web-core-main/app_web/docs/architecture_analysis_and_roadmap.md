# 系统架构分析与演进路线图

## 1. 当前架构分析

### 1.1 现有技术栈

```
┌─────────────────────────────────────────────────────────────┐
│                      前端层 (Frontend)                        │
│  HTML + Jinja2 + Bootstrap + Vanilla JS + Chart.js          │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                      应用层 (Application)                     │
│  Flask (单体应用) + Blueprint 模块化                          │
│  - mesapp.py (主应用，6000+ 行)                               │
│  - photo_api.py, test_report_api.py (蓝图模块)               │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                      数据层 (Data)                            │
│  SQLite + JSON 文件 + NAS 文件系统                            │
│  - web_users.db, test_reports.db                             │

└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                      外部集成 (External)                      │
│  群晖 DSM API (用户认证)                                      │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 当前架构优点

| 优点 | 说明 |
|------|------|
| ✅ 简单直接 | 单体应用易于部署和调试 |
| ✅ 模块化初具雏形 | Blueprint 分离了部分功能 |
| ✅ 轻量级 | 依赖少，资源占用低 |
| ✅ 快速迭代 | 修改即生效，开发效率高 |

### 1.3 当前架构瓶颈

| 问题 | 影响 | 严重程度 |
|------|------|----------|
| ❌ 单体应用过大 | mesapp.py 6000+ 行，难以维护 | 🔴 高 |
| ❌ SQLite 并发限制 | 多设备同时写入会锁表 | 🔴 高 |
| ❌ 缺乏消息队列 | 无法处理设备实时数据流 | 🔴 高 |
| ❌ 无服务治理 | 无法水平扩展、无监控告警 | 🟡 中 |
| ❌ 前端技术陈旧 | Jinja2 模板难以构建复杂交互 | 🟡 中 |
| ❌ 缺乏数据建模 | 各模块数据孤岛，难以关联分析 | 🔴 高 |

## 4. 成熟框架推荐

### 4.1 MES/MOM 开源框架对比

| 框架 | 语言 | 特点 | 适合场景 | 推荐度 |
|------|------|------|----------|--------|
| **OpenMES** | Python | 轻量级、易扩展 | 中小型制造企业 | ⭐⭐⭐⭐ |
| **Apache OFBiz** | Java | 全功能 ERP+MES | 大型企业 | ⭐⭐⭐ |
| **Odoo** | Python | 模块化、生态丰富 | 通用型 | ⭐⭐⭐⭐⭐ |
| **ERPNext** | Python | 开箱即用 | 中小型企业 | ⭐⭐⭐⭐ |
| **QMS (Quality)** | 多种 | 专注质量管理 | 质量为核心 | ⭐⭐⭐ |

### 4.3 为什么推荐 Odoo？

| 优势 | 说明 |
|------|------|
| **开源免费** | 社区版完全免费，企业版可选 |
| **Python 技术栈** | 与当前代码兼容，团队无需学习新语言 |
| **模块化设计** | 需要什么装什么，灵活组合 |
| **MRP/MES 模块成熟** | 制造、质量、库存模块开箱即用 |
| **金蝶集成插件** | 社区有金蝶 K3/Cloud 集成模块 |
| **REST API 完善** | 方便自研采集层对接 |
| **报表功能强** | 内置 BI 报表和仪表板 |

## 9. 结论

### 当前架构评估
- **短期可用**：当前 Flask 单体架构可以继续支撑基本需求
- **中期瓶颈**：设备接入、实时处理、ERP 集成将遇到明显瓶颈
- **长期不可行**：无法支撑 AI 分析、大数据处理、分布式设备管理

### 推荐路径
1. **渐进式演进**：不推荐大规模重写，应分阶段升级
2. **引入 Odoo**：作为业务中台，复用成熟的 MES/QMS 模块
3. **保留自研采集层**：当前 Python 代码可复用，专注设备接入
4. **数据中台优先**：统一数据模型是后续所有功能的基础

### 下一步行动
1. 评估 Odoo 部署和定制工作量
2. 设计统一的产品数据模型
3. 实现 Modbus 采集服务原型

## 11. 阶段一实施方案（已完成）

### 11.1 部署架构

**目标**: Web 服务器与 NAS 存储分离，实现存储与计算分离。

```
┌─────────────────────────────────────────────────┐
│  生产服务器 (172.16.30.X)                        │
│  ├── Docker Compose                             │
│  │   ├── Nginx (反向代理)                        │
│  │   ├── Flask App (应用服务)                    │
│  │   └── Redis (会话缓存)                        │
│  ├── 本地数据 (SSD)                              │
│  │   ├── SQLite 数据库                           │
│  │   ├── 日志文件                                │
│  │   └── 缓存数据                                │
│  └── NAS 挂载点 (网络挂载)                       │
│      └── → 群晖 NAS (172.16.30.2)               │
│          ├── 照片存储 (50GB+)                    │
│          ├── 配置文件 (JSON)                     │
│          └── 记录文件 (CSV/Excel)                │
└─────────────────────────────────────────────────┘
```

### 11.3 部署步骤概要

1. **服务器环境准备** (30分钟)
   - 安装 Ubuntu 22.04 LTS
   - 安装 Docker 和 Docker Compose
   - 配置防火墙和网络

2. **NAS 挂载配置** (15分钟)
   - 运行 `mount-nas.sh` 脚本
   - 验证 NAS 连通性
   - 配置开机自动挂载

3. **应用部署** (30分钟)
   - 克隆代码到服务器
   - 配置 `.env` 环境变量
   - 运行 `deploy.sh` 脚本

4. **数据迁移** (30分钟)
   - 迁移 SQLite 数据库
   - 验证 NAS 数据访问
   - 测试应用功能

5. **生产切换** (30分钟)
   - 更新客户端访问地址
   - 监控系统运行
   - 准备回滚方案

### 11.4 关键改进

| 改进项 | 当前状态 | 新架构 | 效果 |
|-------|---------|--------|------|
| 部署位置 | 群晖 NAS | 独立服务器 | 性能提升 3-5x |
| 数据库 | NAS 存储 | 服务器 SSD | 并发能力 +10x |
| 会话管理 | Flask 默认 | Redis | 支持多实例 |
| 照片访问 | 本地读取 | 网络挂载 + 缓存 | 灵活性提升 |
| 可扩展性 | 单点 | 支持水平扩展 | 可添加负载均衡 |
