# Role Permission Change

这个 skill 用于角色和权限相关需求，尤其适合“新增一个 role”这类容易被误简化成字段改动的任务。

## 何时使用

当满足以下任一条件时使用：

- 新增 role
- 修改角色权限范围
- 修改菜单或页面访问权限
- 修改路由守卫
- 修改按钮级权限
- 修改后台分配角色逻辑
- 修改接口鉴权范围

## Phase 1：需求理解

先把一句模糊需求转成明确目标。

必须输出：

- 角色名称是否已明确
- 该角色的业务职责是什么
- 这是新增功能还是规则调整
- 当前已知信息和未知信息

## Phase 2：需求完善

必须主动补齐以下内容，不能只问“要什么权限”：

- 角色名称
- 角色来源：后台分配、注册选择、外部同步
- 页面访问范围
- 操作权限范围：查看、编辑、删除、导出、审批、管理用户
- 与现有角色关系
- 默认 fallback 行为
- 不允许破坏的旧角色行为
- 前后端是否都需要同步

推荐输出模板：

```md
基于当前信息，我建议将需求补充为：

- 新角色：
- 角色职责：
- 可访问页面：
- 禁止访问页面：
- 可执行操作：
- 禁止操作：
- 分配方式：
- 不允许破坏的旧行为：
- 仍需确认的点：
```

## Phase 3：风险确认

必须评估：

- 菜单可见性与路由守卫是否一致
- 前端按钮权限与后端接口鉴权是否一致
- token/session/user profile 中 role 字段兼容性
- 历史用户数据和缓存兼容性
- 后台角色分配、筛选、展示是否同步
- 未知 role 的 default/fallback 是否会误判

## 必须停手确认的情况

遇到以下任一情况，停止实现：

- 新 role 的页面访问范围不明确
- 新 role 的操作边界不明确
- requirement 未记录现有角色语义
- 前后端权限规则存在冲突
- 本次改动需要触达 auth core、session、route guard、global store，但尚未做边界确认

## Phase 4：实施路径

优先采用最小影响路径，推荐顺序：

1. 回看 requirement 和当前角色定义
2. 确认角色职责与旧角色边界
3. 梳理前端页面、菜单、按钮、路由、后端接口影响面
4. 只在必要位置新增 role 支持
5. 验证新角色和至少一个旧角色
6. 回写 requirement

## 最低验证清单

至少验证：

- 新 role 是否只能访问允许页面
- 新 role 是否被正确禁止访问限制页面
- 新 role 是否只能执行允许操作
- 现有 admin 行为是否不变
- 现有普通角色行为是否不变
- 刷新页面后 role 行为是否保持正确
- fallback/default 是否可见且不会静默越权或静默降权
