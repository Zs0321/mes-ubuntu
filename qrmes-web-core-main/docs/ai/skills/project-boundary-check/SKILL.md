# Project Boundary Check

这个 skill 用于把“先画施工区再动手”变成固定流程。

## 何时使用

当满足以下任一条件时使用：

- 需求刚进入分析阶段
- 需求描述只说“修一下”“加个功能”但未说明影响范围
- 变更可能触达 auth/session、共享 hooks/services、全局状态、配置、数据库、部署链路
- 工程师尚未明确哪些模块允许修改

## 输出顺序

先给出一版边界草案，再决定是否进入需求细化或实现。

输出模板：

```md
## Phase 0：改动边界确认

本次允许修改：
- 

本次尽量不修改：
- 

本次未经确认禁止修改：
- 

这样划分的原因：
- 

如果后续必须跨边界修改，我会先暂停并重新做风险确认。
```

## 默认高风险区

以下区域默认视为高风险，除非工程师已明确授权：

- 鉴权 / session
- 全局状态
- 共享 hooks / services
- 运行时配置读取
- 数据持久化和 schema
- 路由守卫
- 部署脚本和生产路径

## 必须停手确认的情况

遇到以下任一情况，不得继续实现：

- 需要从局部页面改到共享基础设施
- 需要修改 production path、真实 env、默认服务地址
- 需要改变 fallback/default 行为
- 需要修改数据库迁移或已上线 schema
- requirement 没有记录这块行为且需求存在多种业务解释

## 验证要求

边界确认完成后，后续实施仍必须验证：

- 新行为是否符合目标
- 同一流程中至少一个旧行为是否保持正常
- 是否有因跨边界产生的隐藏回归
