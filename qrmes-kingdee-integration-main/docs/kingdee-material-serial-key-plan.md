# 金蝶后台新增“物料编码+序列号组合键”实施方案

更新时间：2026-04-23
适用账套：20260330测试
目标系统：金蝶云星空 / QRMES 金蝶同步中台

## 1. 目标

针对类似物料编码：
- `T45150801.A1`

在金蝶后台提供一个正式可见的“物料编码 + 序列号”组合字段，用于：
- 列表展示
- 追溯查询
- 打印/导出
- 跨岗位统一识别单件对象

建议生成效果：
- `T45150801.A1#SN001`
- 或 `T45150801.A1#序列号`

本方案不建议直接覆盖原始物料编码字段，而是新增一个独立字段。

## 2. 当前基础事实

当前已确认：
- 序列号主档菜单：`XLHZD`
- 真实对象：`BD_SerialMainFile`
- 当前已可通过中台拉取：
  - `FNumber` -> 序列号
  - `FMaterialId.FNumber` -> 物料编码
- 本地中台已打通数据集：
  - `serial_master`
- 当前桥接追溯已用到：
  - `serial_no`
  - `material_code`

说明：
金蝶原始对象里已经有“序列号 + 物料编码”的关系，因此新增字段不是为了补缺失关系，而是为了提供统一展示 / 检索 / 打印键。

## 3. 推荐方案

### 3.1 推荐字段类型
建议新增：
- 中文名：`物料序列组合键`
- 字段标识建议：`F_MES_MaterialSerialKey`

建议作为：
- 金蝶 BOS 自定义字段
- 优先放在序列号主档对象 `BD_SerialMainFile` 上

### 3.2 推荐生成规则
推荐规则：
- `物料编码 + '#' + 序列号`

示例：
- `T45150801.A1#SN001`
- `Leviticus-2In1-LiGFL-WD30-A#2In1LiGFLWD30A2026033000018`

不建议：
- 直接覆盖 `FMaterialId.FNumber`
- 直接把序列号拼回原始物料编码字段

原因：
- 物料编码是主数据口径
- 序列号是单件唯一标识
- 混写会污染原始业务字段语义

## 4. 挂载对象建议

### 方案 A：挂在序列号主档 `BD_SerialMainFile`（首选）
适用：
- 主要用于单件追溯
- 主要围绕序列号主档、质量追溯、打印标签展开

优点：
- 与序列号一一对应
- 最贴近当前中台已打通对象
- 后续中台校验与同步最容易对齐

缺点：
- 如果部分场景不走序列号主档，而直接从其他单据查看，可能还要额外联查

### 方案 B：挂在相关追溯对象上同步显示
可选对象：
- `QT_LotSNRelation`
- `QT_TraceShow` 的相关展示模型

不建议首轮就做。

原因：
- `QT_LotSNRelation` 当前测试账套返回 0 条
- `QT_TraceShow / QT_TraceFilter / QT_TreeModel` 不是普通单据型 WebAPI对象
- 先在 `BD_SerialMainFile` 做稳，再向追溯视图扩展更合理

## 5. 技术实现建议

### 5.1 首选：显示型 / 公式型字段
适用：
- 先要看效果
- 先要让金蝶页面、列表、打印能看见
- 不要求首轮就持久化存储

优点：
- 风险小
- 不改原始业务字段
- 适合先验证业务是否真的长期需要

缺点：
- 某些场景下筛选/索引能力不如持久字段

### 5.2 第二选择：持久化自定义字段
适用：
- 后续要长期筛选、导出、接口读取
- 需要作为正式业务字段沉淀

优点：
- 可稳定用于筛选、导出、打印
- 更适合长期正式方案

缺点：
- 需要字段发布、权限、布局、回填
- 维护成本高于显示字段

### 5.3 推荐实施顺序
建议分两阶段：

第一阶段：
- 在 `BD_SerialMainFile` 上新增显示型/公式型字段
- 名称：`物料序列组合键`
- 验证页面展示、列表展示、导出展示是否满足需求

第二阶段：
- 若业务确认长期使用
- 再升级为持久化自定义字段 `F_MES_MaterialSerialKey`
- 做历史数据回填与正式账同步

## 6. 使用场景建议

建议在这些地方展示该字段：
- 序列号主档列表
- 序列号主档详情页
- 质量追溯相关列表/树节点显示
- 标签打印模板
- 导出模板

可选用于筛选：
- 按组合键直接搜索
- 按序列号搜索
- 按物料编码搜索

## 7. 与当前中台的关系

当前中台已经可以提供本地等价能力：
- `serial_master` 中已落 `material_code + serial_no`
- `GET /api/local-db/trace/serial?serial_no=...&material_code=...`

因此建议：
- 金蝶后台字段方案作为“正式业务可视化字段”推进
- 本地中台继续保留桥接能力，作为验证和兜底层

后续如果金蝶正式字段建好，可同步增强中台：
- 拉取 `F_MES_MaterialSerialKey`
- 对外直接返回该字段
- 用于追溯树节点显示一致化

## 8. 风险点

1. 不要覆盖原始物料编码字段
- 原始 `FMaterialId.FNumber` 必须保持标准物料编码口径

2. 不要在第一阶段就全链路扩散
- 先落 `BD_SerialMainFile`
- 再看是否要扩到追溯页、打印模板、报表

3. 测试账和正式账要分别验证
- 测试账先验证页面、公式、导出
- 正式账再按相同步骤实施

4. 若要用于筛选与高频查询
- 显示字段可能不够
- 需要后续升级为持久化字段

## 9. 推荐实施步骤（金蝶后台）

### 第一步：测试账新增字段
对象：
- `BD_SerialMainFile`

字段：
- 中文名：`物料序列组合键`
- 标识：`F_MES_MaterialSerialKey`

规则：
- `FMaterialId.FNumber + '#' + FNumber`

### 第二步：测试账验证
验证项：
1. 序列号主档列表是否可见
2. 序列号主档详情页是否可见
3. 示例编码如 `T45150801.A1` 对应序列号是否能正确拼接
4. 导出是否带出
5. 打印模板是否可引用

### 第三步：决定是否升级为持久字段
判断标准：
- 是否需要筛选/索引
- 是否需要接口稳定读取
- 是否要成为正式业务口径

### 第四步：正式账迁移
迁移时需同步：
- 字段定义
- 布局
- 权限
- 打印模板
- 导出模板
- 历史数据回填策略

## 10. 最终建议

当前推荐决策：
- 先在金蝶测试账的 `BD_SerialMainFile` 上新增 `F_MES_MaterialSerialKey`
- 第一阶段优先做显示/公式型实现
- 验证通过后再决定是否升级为持久化正式字段

不推荐：
- 直接改原始物料编码字段
- 第一阶段就同时改多个对象
- 在尚未验证业务长期价值前大范围扩散到正式账

## 11. 中台配合建议（后续）

当金蝶后台字段上线后，建议中台追加两项工作：
1. 在 `serial_master` 拉取中增加该字段
2. 在追溯桥接接口中优先返回该字段作为统一显示键

建议接口增强方向：
- `GET /api/local-db/trace/serial`
- `GET /api/local-db/serial_master`

统一返回：
- `material_serial_key`
- `material_code`
- `serial_no`

这样金蝶前台、中台、MES 展示口径可以统一。
